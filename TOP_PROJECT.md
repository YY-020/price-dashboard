# 市场智能体 · 项目宪法

## 一、项目定义

**项目名称**：市场智能体

**一句话Slogan**：锂电行业的"贾维斯"（Jarvis），从数据采集到战略研判的市场部核心基础设施。

**核心价值**：将重复劳动交给机器（效率），将信息过滤与逻辑研判交给AI（质量），让人专注于决策（价值）。

## 二、长期愿景

### 理想工作状态

周五下午3:00，打开企业微信，Agent在上午9:00已自动推送本周周报草稿。花30分钟审核修改，一键发送。

**日常工作中**：
- 同事打开网址，输入账号密码，查看价格走势图、下载数据、查询数据库
- 领导问"碳酸锂最近怎么样"，打开看板，10秒出图
- 在企业微信里 @市场小秘 "查一下LGES最近有什么新闻"，Agent自动返回筛选后的结果
- 问Agent"碳酸锂为什么大涨10%"，它像专家一样拆解问题、查知识库、搜新闻、给出逻辑严密的分析
- 让Agent做硫化锂市场调研，它自动生成大纲、填充二手信息、列出一手调研问题清单等你补充

**不再做的事情**：
- 手动翻10个官网+6个公众号找新闻
- 手动筛选、分类、写摘要
- 手动算涨跌幅、截图、排版
- 周五焦虑赶工到17:30

**专注于**：
- 审核AI的输出，做人类判断
- 积累行业洞察，形成市场观点
- 扩展数据库，持续优化Agent能力

## 三、核心架构

**三大系统 + 一个交互层**：

| 系统 | 功能 | 一句话 |
|:---|:---|:---|
| 系统一：全自动周报/日报流水线 | 数据+新闻→筛选→分析→成文→分发 | 每日自动生成日报草稿 |
| 系统二：专家级研判与调研Agent | 带着脑子拆解问题、自主调研 | 碳酸锂为什么涨？硫化锂市场多大？ |
| 系统三：云端动态数据库 | 多库管理+动态修正+全链路溯源 | 同事登录就能查，数据来源可追溯 |
| 交互层：无处不在的入口 | 企业微信/钉钉/飞书 @就能用 | 不用命令行，傻瓜式交互 |

## 六、技术决策记录

| 决策 | 内容 | 原因 |
|:---|:---|:---|
| 技术栈 | Python + Streamlit + Plotly | 免费、100%可控、可扩展、后期可接入AI Agent |
| 开发工具 | 从 Python+DeepSeek+VS Code 迁移到 Trae | 零门槛、自动写文件、自己调试 |
| 搜索方案 | 放弃 DDG 全网搜索，改用固定站点 RSS + requests | DDG中文噪音极大，RSS稳定可靠 |
| Benchmark渠道 | 删除 | Electrive已覆盖其高价值新闻 |
| 部署方案 | 短期本地+内网，中期云服务器 | 成本低、够用 |
| 数据流 | ETL原则：清洗与展示分离 | 数据问题在源头解决，看板代码不反复修 |
| 价格预测 | 不走纯数据模型 | 基于行业知识+供需逻辑的AI推理 |
| 办公提效工具 | 千问语音、豆包等单独管理，不纳入主线 | 它们是瑞士军刀，不是主战场 |

### 代码分层原则

| 模块 | 只做什么 | 不做什么 |
|:---|:---|:---|
| app.py | Streamlit 编排（读参数→调loader→调chart→显示） | 不读文件、不算数据 |
| data_loader.py | 读CSV→校验→返回DataFrame | 不画图、不处理UI |
| chart_builder.py | 收数据→生成Plotly fig | 不读文件、不关心sidebar |
| constants.py | 存配置（分组、颜色、阈值） | 不写逻辑 |

### 文件命名规范

- 下划线命名法：`data_loader.py`、`chart_builder.py`
- 日期后缀格式：`YYYYMMDD`（如 `price_data_0513.xlsx`）
- 备份文件：`_backup/` 或 `_ARCHIVE/`

### Git 监控范围

- ✅ 监控：`01_Build/` 代码、根目录 `.md` 文件、`.gitignore`
- ❌ 不监控：`00_Inbox/` 数据文件、`02_Output/` 输出、`_DOCS/` 参考文件、`logs/`

## 对话行为约定

- 在执行任何代码、文件或系统操作前，必须先向用户说明：你的理解、你的目标、你的方案。
- 用户确认后，你才能开始执行。
- 不要仅凭一句话指令就自顾自地开始写代码或修改文件。

### 对话风格约定

- 先给结论和整体框架，再展开细节
- 用表格做对比，用层级结构展示逻辑关系
- 修改代码前，先说清楚"改什么、为什么改、影响范围"

### 任务完成度透明原则

- **禁止默默部分完成**：如果因技术限制（如文件大小限制、编码问题、API限制）导致任务无法100%完成，必须明确告知用户
- **必须报告的内容**：
  1. 任务实际完成情况（已完成百分比）
  2. 未完成部分的具体原因
  3. 建议的解决方案或替代方案
- **示例**："文档转换完成80%，剩余20%因单文件读取限制被截断。解决方案：分块读取后合并，预计额外耗时2分钟，是否继续？"

### 对话框管理规则

- 每个新子任务开一个新对话框
- 每个新对话框启动时，必须先读取 `TOP_PROJECT.md` + `TOP_CONTEXT.md` + `工作日志.md`
- 子任务的 `CONTEXT.md` 只写本模块相关内容，不重复TOP文件已有信息
